# 多 Agent 调度操作指南

## 核心原理

Claude Code 中，多个 Agent 不能直接相互对话。**主 Agent（你对话的那个）充当协调中枢**，负责启动、串联、判断。

```
你说一句话
  │
  ▼
主 Agent（协调者）
  │
  ├─► 改代码 Agent（后台运行）──► 完成后回来报告
  │
  ├─► 审查 Agent（后台运行）──► 审查结果回来报告
  │
  └─► 修复 Agent（后台运行）──► 根据审查意见修复
```

---

## 操作方式

### 基础用法：一句话触发

直接在对话中说清楚三件事：
1. **按什么改** — 哪份文档的哪个部分
2. **改什么** — 目标文件
3. **改完后做什么** — 审查、修复等

**示例：**

```
帮我按照 docs/规划/11-前端重构改进方案.md 阶段二的要求，
修改 wexin-read-mcp-main/src/templates/index.html，
改完后自动审查，有问题就继续改直到通过
```

主 Agent 会自动：
1. 启动一个后台 Agent 去改代码
2. 改完后启动一个后台 Agent 审查
3. 审查不过就再启动一个 Agent 修复
4. 最多循环 3 轮后停下来汇报

---

### 进阶用法 1：并行任务

当有多个互不相关的任务时，可以同时启动多个 Agent：

```
同时做两件事：
1. 按 docs/规划/11-前端重构改进方案.md 阶段二改前端
2. 按 docs/规划/10-总任务计划.md 检查后端 API 路由完整性
都完成后统一审查
```

主 Agent 会同时启动两个改代码 Agent，等都完成后启动审查 Agent。

---

### 进阶用法 2：定时监控

设置定时任务，自动检查代码变更：

```
帮我设置一个定时任务，每 10 分钟检查一次项目代码变更，
有变更就告诉我变更了哪些文件
```

主 Agent 会用 Cron 创建定时任务，后台持续监控。

---

### 进阶用法 3：分步执行

如果不确定一步到位，可以分步来：

```
第一步：先帮我读一下 docs/规划/11-前端重构改进方案.md，
告诉我阶段二具体要改什么

第二步：（看完结果后说）好的，按这个改，改完审查
```

---

## 改代码的依据

改代码 Agent 默认按 `docs/规划/` 目录下的文档执行，常用的有：

| 文档 | 内容 |
|------|------|
| `11-前端重构改进方案.md` | 前端 5 阶段重构（安全→内存→代码→CSS→工程化） |
| `10-总任务计划.md` | 9 个已完成任务 + API 路由总览 |
| `05-交易感知系统计划.md` | 交易感知系统实施计划 |
| `04-数据持久化计划.md` | 数据持久化方案 |

你也可以在指令中指定其他文档。

---

## 审查标准

审查 Agent 默认检查：

1. **符合性** — 是否按文档要求修改
2. **安全性** — 有无 XSS、注入等安全问题
3. **逻辑性** — 有无逻辑错误或遗漏
4. **一致性** — 代码风格是否与项目一致

---

## 循环机制

```
第 1 轮：改代码 ──► 审查 ──► 不通过
第 2 轮：修复 ──► 审查 ──► 不通过
第 3 轮：修复 ──► 审查 ──► 通过/不通过 都停止
         │
         └─► 输出最终审查报告，由你决定下一步
```

最多循环 3 轮，避免死循环。3 轮后主 Agent 会停下来告诉你当前状态。

---

## 常见场景速查

| 想做的事 | 怎么说 |
|----------|--------|
| 按文档改代码并审查 | "按文档 XX 改 XX 文件，改完审查" |
| 只审查不改代码 | "帮我审查当前未提交的代码变更" |
| 只改代码不审查 | "按文档 XX 阶段二改前端" |
| 并行改多个文件 | "同时改 A 和 B，都改完审查" |
| 定时监控 | "设置每 10 分钟检查一次代码变更" |
| 停掉定时监控 | "停掉定时监控" |

---

## 注意事项

1. **指令要具体** — 说清「按什么改 + 改什么 + 改完做什么」
2. **大任务拆小** — 一次改太多容易出问题，按文档阶段来
3. **审查结果要读** — 循环结束后看一下审查报告
4. **后台 Agent 有延迟** — 后台 Agent 完成后会自动回调，不需要你等待
5. **最多 3 轮循环** — 如果 3 轮还没过，说明可能需要调整方案，而不是继续修
