## 第一部分：设计规划

### 背景

文章采集存在两条路径：WebSocket 手动交互采集和 Scheduler 定时采集。当前只有 Scheduler 路径会将文章持久化到数据库，WebSocket 路径抓取的文章仅存在于内存中，会话结束后丢失，导致文章管理页面无法看到手动采集的文章。

此外，两条路径各有一套独立的抓取+存储逻辑，存在代码重复和行为不一致的问题。

### 目标

1. 修复 WebSocket 路径不持久化文章的 bug
2. 抽取统一存储层，消除两条路径的重复逻辑
3. 用户可在 WebSocket 采集后勾选要存储的文章（过滤广告）
4. 确保所有改动不破坏现有功能（定时采集、文章管理、报告生成）

### 设计

#### 1. 新增 `src/article_storage.py` — 统一存储服务

所有文章持久化的唯一入口。

##### `save_articles(articles, blogger_id, ai_mentions, skip_existing)`

```python
async def save_articles(
    articles: list[dict],
    blogger_id: str = "",
    ai_mentions: dict[str, list] | None = None,
    skip_existing: bool = True,
) -> dict:
```

- `articles`：`scraper.fetch_article()` 返回的 dict 列表，需包含 `url, title, author, publish_time, content, cover_url`
- `blogger_id`：博主 ID，手动 URL 采集时为空串
- `ai_mentions`：`{url: [mention, ...]}` 按文章 URL 分组的 AI 提及数据，可选
- `skip_existing`：`True` 则 `INSERT OR IGNORE`（按 URL 去重），`False` 强制写入
- 返回：`{"success": True, "inserted": N, "skipped": N}`

内部逻辑：
1. 如果 `skip_existing=True`，查询已有 URL 集合过滤
2. 遍历 articles，组装 INSERT 语句写入 `scraped_articles` 表
3. 返回插入/跳过的计数

##### `get_existing_urls(blogger_id="") -> set[str]`

获取已存储的文章 URL 集合，用于去重检查。

##### `update_mentions(url, mentions) -> bool`

更新指定文章的 `ai_mentions` 字段（用于 Scheduler 路径先存文章后更新 mentions 的场景）。

#### 2. 修改 WebSocket `/ws/task` 流程（`routers/blogger.py`）

在 `websocket_task` 函数中，阶段 1.5（AI 扫描）和阶段 2（报告生成）之间插入勾选步骤。

改动后流程：

```
阶段1:   抓取文章 (_scrape_urls)          ← 不变
阶段1.5: AI 股票提及扫描                 ← 不变
阶段1.6: 【新增】发送文章列表给前端勾选
  → 前端展示标题、作者、发布时间、内容摘要
  → 默认全部勾选，用户可取消广告
  → WebSocket 等待用户回传 selected_urls
阶段2:   save_articles(selected) → 生成报告   ← 用统一存储层
阶段3:   发送邮件/结束                      ← 不变
```

具体改动：
- 在约 498 行 `articles_collected` 消息中新增 `articles` 字段，包含每篇文章的摘要信息
- 新增 WebSocket 等待接收 `selected_urls` 的逻辑
- 调用 `save_articles(selected_articles, blogger_id="", ai_mentions=mentions_by_url)`
- 如果用户未选中任何文章（`selected_urls` 为空），跳过存储直接进入报告
- 前向兼容：如果前端未发送 `selected_urls` 字段（旧版本），默认全部存储

`_scrape_urls` 需要额外返回 blogger_id 映射，以便存储时关联博主。

#### 3. 修改 Scheduler `auto_scrape_job`（`scheduler.py`）

将手写的去重 + INSERT 逻辑替换为调用统一存储层。

改动后流程：

```
获取博主列表 → 遍历博主 → fetch_recent_articles
→ get_existing_urls 去重 → save_articles(全部)
→ AI 扫描 → update_mentions
```

具体改动：
- 删除 scheduler.py 手写存储逻辑（以函数名定位，勿依赖行号）
- 用 `save_articles()` + `update_mentions()` 替代
- AI mentions 处理变为：先存文章（`ai_mentions=None`），扫描完成后调用 `update_mentions` 更新

#### 4. 前端勾选交互（`templates/index.html`）

改动后流程：

```
用户点"开始收集" → WebSocket → 抓取进度
→ 【新增】显示文章勾选列表（默认全选）
→ 用户取消广告，点"确认收集"
→ 存库 → 报告预览 → 发邮件/结束
```

具体改动：
- WebSocket `onmessage` 处理 `articles_collected` 消息时，切换到勾选模式
- 渲染文章列表：每行 checkbox + 标题 + 作者 + 发布时间 + 内容摘要
- 底部提供"全选/取消全选" + "确认收集"按钮
- 点击"确认收集"后读取勾选 URL，WebSocket 发送 `{"selected_urls": [...]}`
- 切回进度条/报告模式

后端消息格式变更：`articles_collected` 消息新增 `articles` 数组字段，每项含 `url, title, author, publish_time, content_preview`。

### 🔴 严重问题

#### 问题 1：WebSocket 等待 `selected_urls` 没有超时和断线处理

阶段 1.6 后端 `await ws.receive_json()` 等待用户勾选，但规划中没有说明：

- 用户打开勾选界面后直接关掉浏览器 → WebSocket 断开，`receive_json()` 抛异常，静默失败
- 用户一直不点"确认收集" → 后端永久挂起，Semaphore 不释放

**修复建议**：

```python
try:
    msg = await asyncio.wait_for(ws.receive_json(), timeout=300)
    selected_urls = msg.get("selected_urls", None)
except asyncio.TimeoutError:
    selected_urls = None  # 超时则默认全部存储
except WebSocketDisconnect:
    return  # 断线直接退出，不存储
```

**规划需补充**：超时时的默认行为（全存 or 跳过）需明确写入设计。

#### 问题 2：两条路径的 AI 扫描时序不一致，形成双重标准

| 路径 | AI 扫描时序 |
|------|------------|
| WebSocket | 先扫描（阶段1.5），后存储（阶段1.6 确认后调 `save_articles(ai_mentions=mentions)`） |
| Scheduler | 先存储（`save_articles(ai_mentions=None)`），后扫描，再 `update_mentions` |

两条路径入库时 `ai_mentions` 的状态不同：WebSocket 路径入库即有值，Scheduler 路径入库时为 `NULL`，需等第二次 `update_mentions` 后才有值。

**问题**：文章管理页若依据 `ai_mentions IS NOT NULL` 判断"已扫描"状态，两条路径的文章会显示不一致。此外，`NULL`（未扫描）和 `[]`（扫描后无提及）在当前表结构中无法区分，建议在 `scraped_articles` 表中增加 `scan_status TEXT DEFAULT 'pending'`（`pending` / `done`）字段做明确区分。

**规划需补充**：明确两条路径的 `ai_mentions` 入库状态，或在表中增加扫描状态字段。

### 🟠 高优先级问题

#### 问题 3：`_scrape_urls` 签名变更是破坏性改动，规划未定义新格式

> "_scrape_urls 需要额外返回 blogger_id 映射，以便存储时关联博主。"

规划未说明：
- 当前返回值结构是什么
- 改动后返回格式（元组 `(articles, blogger_url_map)` 还是 articles 内每个 dict 新增 `blogger_id` 字段）
- 是否有其他调用方

**建议**：优先在每个 article dict 内直接附加 `blogger_id` 字段，而不是返回额外的映射 dict，这样调用方无需解包元组，兼容性更好。

#### 问题 4：`save_articles()` 错误契约未定义

接口返回 `{"success": True/False, ...}`，但未说明：
- DB 写入报错时，返回 `{"success": False, "error": "..."}` 还是直接抛出异常？
- 部分写入失败（前 N 条成功，第 N+1 条报错）时是否回滚？`inserted` 计数如何计算？

调用方（WebSocket 流程和 Scheduler）行为完全依赖此契约，未定义会导致各自假设不同。

**建议**：明确写入"遇到 DB 错误时抛出异常，由调用方负责 catch 并向前端发送错误消息"，比返回 `success=False` 更清晰。

#### 问题 5：Scheduler 行号引用极可能已过期

> "删除 scheduler.py 第 70-135 行的手写存储逻辑"

M2 最新提交（`35a70bb`）刚完成定时采集逻辑，行号大概率已变。

**建议**：改为以函数名定位，例如"删除 `auto_scrape_job` 函数中从 `for article in new_articles:` 到 `logger.info(...)` 的手写 INSERT 段落"。

---

### 🟡 中等问题

#### 问题 6：`content_preview` 长度未定义

后端 `articles_collected` 消息中有 `content_preview` 字段，但截取长度（100字？200字？）和方式（按汉字截取？避免截断标签？）未规定，前端渲染和后端生成会各自假设。

**建议**：固定为"正文前 150 个汉字，过滤 HTML 标签后截取"，在 `article_storage.py` 中提供统一的 `make_preview(content, length=150)` 工具函数。

#### 问题 7：`blogger_id=""` 写入 `NOT NULL` 字段存在语义歧义

`scraped_articles.blogger_id TEXT NOT NULL`，空字符串技术合法，但：
- 文章管理页按博主筛选时，空字符串博主会形成隐式分组
- `get_existing_urls(blogger_id="")` 可能与数据异常的博主（`blogger_id` 也为空串）污染去重集合

**建议**：手动 URL 采集统一使用 `blogger_id="__manual__"` 而非空串。

#### 问题 8：验证方式缺少异常路径和并发场景

当前验证方式只覆盖 happy path，建议补充：
- WebSocket 中途断开后重新连接，是否产生重复文章
- Scheduler 与 WebSocket 同时运行，同一 URL 的并发 `INSERT OR IGNORE` 测试
- `selected_urls=[]`（全不选）时，报告是否仍正常生成

### 🔵 低优先级建议

#### 问题 9：前向兼容的触发条件需要更精确

> "如果前端未发送 `selected_urls` 字段，默认全部存储"

"旧版前端未发送"与"新版前端用户一篇都不选"在协议层表现相同，需在实现中明确区分：

```python
if "selected_urls" not in data:
    # 旧版前端：默认全部存储
    selected = all_articles
elif data["selected_urls"] == []:
    # 新版前端：用户主动不选，跳过存储
    selected = []
else:
    selected = [a for a in all_articles if a["url"] in set(data["selected_urls"])]
```
