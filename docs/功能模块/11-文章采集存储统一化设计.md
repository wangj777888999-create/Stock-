# 文章采集存储统一化设计

## 背景

文章采集存在两条路径：WebSocket 手动交互采集和 Scheduler 定时采集。当前只有 Scheduler 路径会将文章持久化到数据库，WebSocket 路径抓取的文章仅存在于内存中，会话结束后丢失，导致文章管理页面无法看到手动采集的文章。

此外，两条路径各有一套独立的抓取+存储逻辑，存在代码重复和行为不一致的问题。

## 目标

1. 修复 WebSocket 路径不持久化文章的 bug
2. 抽取统一存储层，消除两条路径的重复逻辑
3. 用户可在 WebSocket 采集后勾选要存储的文章（过滤广告）
4. 确保所有改动不破坏现有功能（定时采集、文章管理、报告生成）

## 设计

### 1. 新增 `src/article_storage.py` — 统一存储服务

所有文章持久化的唯一入口。

#### `save_articles(articles, blogger_id, ai_mentions, skip_existing)`

```python
async def save_articles(
    articles: list[dict],
    blogger_id: str = "",
    ai_mentions: dict[str, list] | None = None,
    skip_existing: bool = True,
) -> dict:
```

- `articles`: `scraper.fetch_article()` 返回的 dict 列表，需包含 `url, title, author, publish_time, content, cover_url`
- `blogger_id`: 博主 ID，手动 URL 采集时为空串
- `ai_mentions`: `{url: [mention, ...]}` 按文章 URL 分组的 AI 提及数据，可选
- `skip_existing`: `True` 则 `INSERT OR IGNORE`（按 URL 去重），`False` 强制写入
- 返回: `{"success": True, "inserted": N, "skipped": N}`

内部逻辑：
1. 如果 `skip_existing=True`，查询已有 URL 集合过滤
2. 遍历 articles，组装 INSERT 语句写入 `scraped_articles` 表
3. 返回插入/跳过的计数

#### `get_existing_urls(blogger_id="") -> set[str]`

获取已存储的文章 URL 集合，用于去重检查。

#### `update_mentions(url, mentions) -> bool`

更新指定文章的 `ai_mentions` 字段（用于 Scheduler 路径先存文章后更新 mentions 的场景）。

### 2. 修改 WebSocket `/ws/task` 流程（`routers/blogger.py`）

在 `websocket_task` 函数中，阶段 1.5（AI 扫描）和阶段 2（报告生成）之间插入勾选步骤。

改动后流程：

```
阶段1:   抓取文章 (_scrape_urls)          ← 不变
阶段1.5: AI 股票提及扫描                 ← 不变
阶段1.6: 【新增】发送文章列表给前端勾选
  → 前端展示标题、作者、发布时间、内容摘要
  → 默认全部勾选，用户可取消广告
  → WebSocket 等待用户回传 selected_urls
阶段2:   save_articles(selected) → 生成报告   ← 用统一存储层
阶段3:   发送邮件/结束                      ← 不变
```

具体改动：
- 在约 498 行 `articles_collected` 消息中新增 `articles` 字段，包含每篇文章的摘要信息
- 新增 WebSocket 等待接收 `selected_urls` 的逻辑
- 调用 `save_articles(selected_articles, blogger_id="", ai_mentions=mentions_by_url)`
- 如果用户未选中任何文章（`selected_urls` 为空），跳过存储直接进入报告
- 前向兼容：如果前端未发送 `selected_urls` 字段（旧版本），默认全部存储

`_scrape_urls` 需要额外返回 blogger_id 映射，以便存储时关联博主。

### 3. 修改 Scheduler `auto_scrape_job`（`scheduler.py`）

将手写的去重 + INSERT 逻辑替换为调用统一存储层。

改动后流程：

```
获取博主列表 → 遍历博主 → fetch_recent_articles
→ get_existing_urls 去重 → save_articles(全部)
→ AI 扫描 → update_mentions
```

具体改动：
- 删除 scheduler.py 第 70-135 行的手写存储逻辑
- 用 `save_articles()` + `update_mentions()` 替代
- AI mentions 处理变为：先存文章（`ai_mentions=None`），扫描完成后调用 `update_mentions` 更新

### 4. 前端勾选交互（`templates/index.html`）

改动后流程：

```
用户点"开始收集" → WebSocket → 抓取进度
→ 【新增】显示文章勾选列表（默认全选）
→ 用户取消广告，点"确认收集"
→ 存库 → 报告预览 → 发邮件/结束
```

具体改动：
- WebSocket `onmessage` 处理 `articles_collected` 消息时，切换到勾选模式
- 渲染文章列表：每行 checkbox + 标题 + 作者 + 发布时间 + 内容摘要
- 底部提供"全选/取消全选" + "确认收集"按钮
- 点击"确认收集"后读取勾选 URL，WebSocket 发送 `{"selected_urls": [...]}`
- 切回进度条/报告模式

后端消息格式变更：`articles_collected` 消息新增 `articles` 数组字段，每项含 `url, title, author, publish_time, content_preview`。

## 文件变更清单

| 文件 | 操作 | 说明 |
|------|------|------|
| `src/article_storage.py` | 新建 | 统一存储服务 |
| `src/routers/blogger.py` | 修改 | WebSocket 流程加入勾选+存储 |
| `src/scheduler.py` | 修改 | 调用统一存储层 |
| `templates/index.html` | 修改 | 前端勾选 UI |

## 验证方式

1. WebSocket 手动采集：抓取文章 → 勾选部分文章 → 确认 → 文章管理页出现选中的文章
2. Scheduler 定时采集：定时任务执行 → 文章管理页出现新文章（行为与改动前一致）
3. 文章管理页：筛选、标记已处理、删除功能正常
4. 去重验证：同一篇文章不重复入库
5. 前向兼容：旧版前端（不发送 selected_urls）默认全部存储
