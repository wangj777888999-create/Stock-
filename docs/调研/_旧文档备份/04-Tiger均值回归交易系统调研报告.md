# Tiger 均值回归半自动交易系统调研报告

> 调研对象：基于 Python + Tiger Trade API 的"价值底仓 + 均值回归"半自动交易系统
> 调研日期：2026-05-09
> 调研目的：评估该方案的设计亮点，提炼可借鉴的设计模式与策略逻辑

---

## 一、方案概述

该系统是一套面向港股的半自动交易辅助工具，核心逻辑分为两层：

- **选股层**：硬性财务指标筛选（PE<20、股息率>4%、ROE>10%、市值>100亿港币）
- **择时层**：MA120 乖离率 + RSI 双重确认（偏离半年线 ±12% 且 RSI 超买/超卖）

执行方式为"机器决策 + 人工下单"的半自动模式，通过飞书 Webhook 推送信号到手机。

---

## 二、可取之处

### 2.1 策略设计：第一性原理的朴素回归

该方案最突出的优点不是技术实现，而是**策略哲学的克制**。

**选股层的四道安全阀：**

| 条件 | 作用 | 设计意图 |
|------|------|---------|
| PE < 20 | 估值锚 | 排除高估值成长股，只买"不贵"的公司 |
| 股息率 > 4% | 现金流验证 | 有真金白银分红，排除"纸面利润" |
| ROE > 10% | 资本效率 | 确保公司能持续创造股东回报 |
| 市值 > 100 亿 | 流动性保护 | 避免小盘股操纵风险 |

四条规则的精妙之处在于**互为补充**：PE 控估值上限，股息验证利润质量，ROE 验证管理能力，市值过滤流动性。这不是单一维度的"低 PE 选股"，而是一个多因子安全边际框架。

**择时层的对称设计：**

- 买入：MA120 × 88%（偏离 -12%）且 RSI < 35
- 卖出：MA120 × 112%（偏离 +12%）且 RSI > 65

±12% 的阈值并非随意设定——港股的年化波动率大约在 15-25%，12% 的偏离约等于半个标准差的运动，是一个"足够异常但不至于极端"的触发点。RSI 作为确认指标，避免在单边下跌趋势中抄底。

### 2.2 架构设计：极简主义

整个系统只有一个 Python 文件（`monitor.py`），约 80 行有效代码。这种"够用就好"的设计哲学值得学习：

**无框架依赖**：不依赖任何量化框架（如 Backtrader、Zipline），仅用 `tigeropen`（数据源）+ `pandas`（计算）+ `requests`（通知）三个库。对于半自动系统来说，去掉回测引擎和订单管理模块是正确的选择。

**无状态设计**：每次运行独立完成"取数据→计算→判断→通知"四步，不依赖数据库、不维护持仓状态。状态管理完全交给交易者本人（手动更新 watchlist、手动下单）。这避免了大量工程复杂度：

- 不需要处理网络断连后的状态恢复
- 不需要担心信号重复发送
- 不需要数据库运维

**单一职责**：脚本只做"信号检测+通知"，不做持仓管理、不做下单执行、不做盈亏统计。这种边界划分让系统极其稳定——它只依赖行情数据的可用性，不依赖交易 API 的可用性。

### 2.3 人机协作：半自动的正确打开方式

该方案没有追求全自动交易，而是设计了一个**"机器筛、人决策"**的协作模式：

```
机器负责：                人负责：
├─ 每日定时扫描            ├─ 每周更新选股池
├─ 计算指标                ├─ 确认信号合理性
├─ 判断信号                ├─ 检查是否有黑天鹅
└─ 推送通知               └─ 手动下单
```

这个设计承认了两个现实：

1. **个人投资者无法 7×24 盯盘** — 机器代替了最枯燥的扫描工作
2. **机器无法理解突发事件** — 人在下单前检查停牌、黑天鹅等机器看不到的上下文

飞书推送的时机（15:30，收盘前 30 分钟）也经过考量——港股 16:00 收盘，30 分钟的窗口足够人工判断和操作。

### 2.4 部署方案：零运维

用 crontab 实现定时执行，而不是用 Docker、systemd 或任何进程管理工具。对于一个每天跑一次、每次跑 5 秒钟的脚本来说，crontab 是最正确的选择。日志重定向到文件（`>> trade.log 2>&1`），够用就行。

## 三、局限性（供参考，非主要调研目标）

| 局限 | 说明 |
|------|------|
| 选股依赖人工 | 每周手动更新 `my_watchlist`，没有自动化财务筛选 |
| 无回测验证 | 没有历史数据验证 MA120±12% + RSI 的胜率和夏普比率 |
| 单市场 | 仅支持港股，不覆盖 A 股或美股 |
| 无复权核对 | 提示了除权可能导致 MA120 位移，但未做自动检测 |
| 通知单通道 | 仅飞书 Webhook，无降级机制 |

## 四、对 StockPulse 的可借鉴价值

### 4.1 策略层面

**均值回归框架可直接复用。** StockPulse 已有 RSI(14) 计算和港股日K数据，只差一个 SMA 函数即可完整实现该策略。±12% 的阈值可以参数化，方便后续做敏感性分析。

**多因子选股思路可扩展。** 我们的 `iwencai_service.py` 支持自然语言选股查询，可以将"PE<20 + 股息>4% + ROE>10% + 市值>100亿"转化为问财查询语句，实现选股自动化。

### 4.2 工程层面

**"信号检测与交易执行分离"的设计模式值得采纳。** 即使我们有模拟交易模块，也应该把信号生成和信号执行解耦——信号是分析结果，执行是用户决策。这与我们已有的 `backtests`（策略记录）+ `sim_trades`（模拟执行）+ `trade_journal`（实盘记录）三层结构天然吻合。

**参数化阈值设计。** 该方案将 MA120 的偏离比例（88%/112%）和 RSI 阈值（35/65）硬编码，但我们可以在实现时将其做成可配置参数，存入 `backtests.params` 字段，方便后续对比不同参数组合的效果。

### 4.3 产品层面

**半自动模式契合个人投资者场景。** 该方案证明了"不追求全自动，做好信号推送+人工确认"是可行的产品定位。我们的目标用户（个人投资者）需要的是辅助决策工具，不是高频交易系统。

**推送通知是关键体验环节。** 信号再准确，用户看不到也白搭。飞书 Webhook 的方案轻量且实时，比邮件更适合交易场景（邮件经常延迟或进垃圾箱）。

---

## 五、总结

这套 Tiger 交易系统的核心价值不在技术实现的精巧，而在于**策略设计的克制和工程取舍的务实**：

1. **用四条财务指标守住安全边际** — 不追热点、不猜涨跌
2. **用 MA120 + RSI 做均值回归择时** — 只在情绪极端时出手
3. **半自动模式承认人的价值** — 机器做计算，人做判断
4. **极简架构保证稳定性** — 80 行代码比 800 行更不容易出 Bug

对于 StockPulse 而言，最直接的价值是**提供了一个可落地的策略模板**——我们已经有数据和指标计算能力，只需补齐策略封装层，即可将这套逻辑整合进现有系统。
