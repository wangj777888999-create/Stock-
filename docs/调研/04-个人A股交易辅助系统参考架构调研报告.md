# 04 - 个人 A 股交易辅助系统参考架构调研报告

> **调研日期**：2026-05-17
> **来源**：公开分享的个人交易系统设计图（社区截图，非开源代码）
> **系统别名**：TradingAgents 个人版

---

## 一、系统定位

**目标用户**：中低频个人 A 股投资者（不做高频/日内，以日线、隔日为主）

**核心理念**：把"选股 → 计划 → 盯盘 → 复盘 → 迭代"变成稳定可执行的流程，而非依赖盘感或随机操作。

**设计原则**：
- 稳定优先 — 宁可漏信号，不做噪音
- 默认静默 — 只在触发阈值时推送，不做状态播报
- 人工+AI 协作 — AI 负责数据聚合与规则判断，人工做最终决策
- 可追溯 — 每次推送同时落盘，便于后续复盘迭代

---

## 二、总体架构

系统由四层构成，单向数据流：

```
信息源 (Data Sources)
    ↓
DataHub（统一数据层 + 缓存落库）
    ↓
智能体系统 (Agents)：主调度 + 策略 A (Joyce) + 策略 B (Brooks)
    ↓
调度与输出 (Scheduler + 飞书机器人)
```

### 2.1 信息源层

| 数据源 | 类型 | 说明 |
|--------|------|------|
| **Tushare Pro** | 结构化历史数据 | 日线/估值/财报/业绩，盘后延时更新 |
| **新浪财经/智研 MCP** | 实时/分钟级 | A/HK/US 实时报价（按额度限流），分钟K可选 |
| **用户手动导入** | 非结构化 | 事件要点、公告摘要、观察清单、仓位初始化 |

**设计取舍**：历史结构化数据走 Tushare（权威性高但有延时），盘中快照走 MCP（实时但按频率），两者在 DataHub 合流，互为补充，不互相依赖。

### 2.2 DataHub 层

**职责**：统一数据入口，屏蔽多源差异，向上层 Agent 提供稳定的 Data API。

| 能力 | 实现方式 |
|------|---------|
| 增量拉取 | 仅拉取缺失/更新数据，不全量重建 |
| 去重与补缺 | 检测缺失日期、异常值、延迟数据 |
| 质量检查 | 对缺失率/异常值/更新时间窗口做门控 |
| 多源降级 | 某源失败时自动切换备源 |
| 落库 | SQLite（轻量）或 DuckDB（分析型），结构统一 |

**输出接口**：上层 Agent 只调 Data API，不感知底层数据源细节，便于替换数据源而不影响分析逻辑。

### 2.3 智能体系统

系统内置三个 Agent，形成"主调度 + 双策略视角"结构：

#### 主调度 Agent（编排者）
- 选择当前场景：Plan / Observe / Review / Iterate
- 接收策略 Agent A/B 的输出，合并为可执行的"一页纸"结论
- 声明数据截止点和不可得项（如分钟盘口、逐笔等）

#### 策略 Agent A — Joyce（VCP 价值龙头趋势，中长线/选股）
- **市场温度计**：判断整体仓位上限
- **赛道白名单 + 龙头筛选** → 候选池 TopN
- **给出盘前执行纪律**，不追盘中噪音

核心关键词：趋势、主线、风控纪律

#### 策略 Agent B — Brooks（裸K价格行为，短线/进出场）
- **结构三态**：趋势 / 区间 / 反转
- **信号**：High1 / High2 及对应止损规则
- **强调尾盘确认**，减少盘中误判

核心关键词：结构位、止损、确认

### 2.4 调度与输出层

| 组件 | 职责 |
|------|------|
| **场景编排器** | 按时间（盘前/盘中/盘后/周末）触发对应 Agent 组合 |
| **飞书机器人** | 接收 Agent 输出，按短报/长报/告警/周报格式推送 |
| **输出原则** | 默认静默，减少信息噪音；只给建议动作/风控边界，不自动执行交易 |
| **本地落盘** | 每次推送同步写入 watchlist / signals / run_logs / data_quality |

## 三、四大场景流程

### 场景 01：盘前计划（Plan）

**触发时机**：开盘前（如早 8:30）

**输入**：
- Tushare 前一交易日收盘数据（日线/估值/财报）
- 持仓 + 观察池（飞书/本地配置：仓位、成本、角色、关注标的与阈值）

**处理流程**：

```
DataHub 增量拉取 + 缓存落库（去重/补缺/质量检查）
    ↓
策略 Agent A (Joyce)：
  市场温度计 → 今日仓位上限
  赛道白名单 + 龙头过滤 → 候选池 TopN
  盘前执行纪律（不追盘中噪音）
    ↓
策略 Agent B (Brooks)：
  趋势/区间/反转三态判断
  High1/High2 信号 + 止损规则
  强调尾盘确认
    ↓
主调度 Agent：
  合并多方结论 → 一页纸
  声明数据截止点 & 不可得项
```

**输出**（飞书推送，短版为主，可选长版）：
1. 今日市场假设（宏观情绪/板块轮动）
2. 候选池 TopN（每只标的 3 条进池理由）
3. 持仓纪律/仓位上限
4. 观察阈值与风险清单

**同时落盘**：watchlist / signals / run_logs / data_quality

---

### 场景 02：盘中盯盘（Observe）

**触发时机**：低频，仅阈值触发，默认静默

**输入**：
- 新浪财经 MCP 实时报价（A/HK/US）
- 盘前输出的观察池 + 阈值 + 持仓纪律

**处理流程**：

```
调度器：按市场时段低频巡检
  A股 开盘/尾盘加密，其余 5-15 分钟
  港/美：更低频（节省额度）
    ↓
规则引擎：只判断"触发/不触发"
  涨跌幅阈值 | 连续异动 | 量能突变 | 重大事件
  可扩展：结合 Brooks/Joyce 关键结构价位
    ↓
降噪模块（硬规则）：
  合并窗口 15 分钟（同标的多次触发合并）
  单日上限 N 条（如 5 条）
  只输出"触发项"，不刷常规状态
```

**输出**（飞书推送，仅触发时）：
- 触发类型 | 标的 | 触发依据（价位/阈值/时间） | 建议动作（关注/降低风险/等收盘确认）
- 同时写入 outbox + intraday 快照（便于盘后复盘）

---

### 场景 03：盘后复盘（Review）

**触发时机**：收盘后（如 15:30）

**目标**：用当日完整日线"把结果说清楚"：发生了什么 → 为什么 → 明天怎么做

**输入**：
- Tushare 盘后更新的当日数据（日线/估值/涨跌分布/资金类）
- 盘中快照（若启用）+ 事件/告警记录

**处理流程**：

```
DataHub 入库 + 派生指标：
  收益/回撤/振幅/趋势强弱/量价结构等
  写入 bars_daily / daily_basic / signals
    ↓
策略复盘 — Joyce + Brooks 双视角：
  Joyce：主线/龙头/仓位纪律是否执行到位
  Brooks：结构三态 + 信号是否确认（尾盘）
  共识输出：哪些结论可延续到明天
    ↓
质量门控：
  若数据缺失/延迟 → 报告降级并标注"仅供参考"
  若盘中误报 → 更新阈值/合并策略/观察池
  把"系统变好"写进流程
```

**输出**（飞书推送，长报为主，可附短报）：
- 市场复盘 | 持仓复盘 | 候选池更新 | 明日计划 | 风险清单（含数据截止点/触发依据）
- 同时落盘：data_quality_report / run_logs / watchlist 更新 / signals

---

### 场景 04：选股与题材挖掘（Discovery）

**触发时机**：周末或行情平淡时段，非每日强制

**目标**：把"全市场噪音"压缩成"可研究的 Top 清单"，并能一键深度分析

**输入**：
- Tushare/DataHub 日线/估值/财务/业绩预告/行业分类
- 偏好与策略约束（赛道白名单/不碰行业/市值与流动性门槛/交易风格）

**三层过滤**：

```
第一层：可交易 & 可研究
  剔除：ST/停牌/流动性差/财务异常
  保留：成交额/市值/行业符合门槛
  → 输出候选池 Candidate Pool

第二层：策略打分（多维度）
  VCP/趋势强度：缩量回调、关键均线、波动收缩
  基本面/预期：业绩预告、估值合理性、确定性信号
  风险：高位放量滞涨、补涨龙、跌破结构位
  → 输出排序后的候选池

第三层：生成"研究任务"
  对 TopN 标的：产出研究提纲/关键问题/验证清单
  可选：一键调用 upstream 深研（生成长报告）
  → 写入 watchlist / signals / 深研入口
```

**输出**（飞书推送：候选池 TopN + 题材/主线摘要）：
- 每只标的给出：为什么在池里（3 条理由）| 关键价位/阈值 | 风险点 | 下一步验证什么
- 同时落盘：候选池快照（可对比昨天/上周），支持持续迭代

## 四、核心设计亮点

### 4.1 降噪优先设计
系统在盘中（Observe）明确内置"降噪模块"，以硬规则限制推送频次：
- 合并窗口 15 分钟：同标的触发合并，避免短时间内重复打扰
- 单日上限 N 条：防止极端行情下信息泛滥
- 只输出触发项：不刷常规状态，保护注意力

### 4.2 数据质量门控
在 Review 场景明确引入质量门控节点：
- 数据缺失/延迟 → 报告自动降级并标注
- 盘中误报 → 触发参数更新（而非静默接受）
- "把系统变好"写进流程 — 质量改进是每日必须项，而非可选项

### 4.3 双视角策略设计
Joyce（趋势/基本面/中长线） vs. Brooks（价格行为/结构/短线）不做合并，各自输出独立结论，最终由主调度 Agent 合并。

这避免了单一策略的盲点，同时保留了两种视角的完整性，让人工决策有更丰富的参考维度。

### 4.4 推送即落盘
每次飞书推送同步写入本地（watchlist / signals / run_logs），保证：
- 可回溯：每个决策都有依据记录
- 可迭代：Discovery 场景的候选池可与上周对比
- 可审计：数据质量问题有明确记录

## 六、与 StockPulse 的对比

### 6.1 理念层面

| 维度 | 参考系统（TradingAgents） | StockPulse |
|------|--------------------------|------------|
| 定位 | 纯个人 A 股，极度精简 | 多市场全功能平台 |
| 推送机制 | 飞书机器人（移动端友好） | Web 前端（桌面为主） |
| 降噪设计 | 显式硬规则（15分钟合并/单日上限） | 无显式降噪规则 |
| 数据质量门控 | 内置质量降级机制 | 无 |
| Agent 编排 | 显式主调度 + 双策略结构 | 3 个人设并行，无主调度 |
| Discovery 场景 | 三层过滤 + 研究任务生成 | iwencai 自然语言选股 |

### 6.2 能力差距

| 差距 | 参考系统有 | StockPulse 现状 |
|------|-----------|----------------|
| 显式降噪规则 | 有（合并窗口+上限） | 无 |
| 盘中实时推送 | 有（飞书触发式） | 无 |
| 数据质量降级报告 | 有 | 无 |
| 场景化编排（Plan/Observe/Review） | 有 | 无 |
| 研究任务自动生成 | 有（Discovery 第三层） | 无 |
| 移动端推送通道 | 有（飞书） | 无 |
| 主调度 Agent | 有 | 无（3 人设独立） |

### 6.3 StockPulse 的优势

| 维度 | StockPulse 优势 |
|------|----------------|
| 市场覆盖 | A/港/美/韩/日/期货/ETF/加密货币 |
| 数据源广度 | 腾讯 + AKShare + yfinance + CoinGecko |
| 前端体验 | 完整 SPA，9 页面，图表/画线/标注 |
| 微信博主 | 完整管理 + Playwright 抓取 + AI 分析 |
| 交易记录 | 模拟盘/实盘/胜率统计完整 |

## 七、可借鉴项与优先级

### P0（高价值、与现有架构高度兼容）

| 借鉴项 | 具体实施 |
|--------|---------|
| **场景化编排（Plan / Observe / Review）** | 在现有 AI 分析流程上增加"场景上下文"，让 3 个人设分析器知道当前是盘前/盘中/盘后 |
| **降噪硬规则** | 为未来的推送功能预设合并窗口 + 单日上限参数，写进 config |
| **推送即落盘** | 现有分析结果已落 DB，但缺少 run_logs / data_quality_report 两类结构化记录 |

### P1（高价值、需要新增模块）

| 借鉴项 | 具体实施 |
|--------|---------|
| **数据质量门控** | 在 DataHub 层（stock_utils.py 缓存体系）增加质量检查：缺失率/异常值/更新时间窗口 |
| **主调度 Agent** | 在 analyzer.py 上层增加编排层，合并 3 个人设输出，生成"一页纸"结论 |
| **Discovery 三层过滤** | 在 iwencai_service.py 基础上，增加 VCP 形态评分 + 基本面确定性信号层 |

### P2（中价值、依赖外部服务）

| 借鉴项 | 具体实施 |
|--------|---------|
| **飞书机器人推送** | 新增 `notification/feishu.py`，接入飞书 Webhook，推送盘前/盘中/复盘报告 |
| **盘中实时巡检** | 利用现有腾讯 API 实时行情，增加定时任务 + 阈值规则引擎 |

## 八、结论

这套参考架构最大的价值不在于技术复杂度（技术栈相对简单），而在于**流程设计的完整性与纪律性**：

1. **完整闭环**：选股 → 计划 → 盯盘 → 复盘 → 迭代，每个场景都有明确的输入/输出/落盘
2. **纪律内化**：降噪、质量门控、执行纪律不依赖人的自律，而是写进系统规则
3. **轻量可执行**：没有过度工程化，以飞书推送为交互界面，极低操作成本

StockPulse 已具备数据采集、AI 分析、模拟交易的能力，但缺少**场景编排层**（什么时候、什么情境、输出什么格式的内容给人看）。引入 Plan/Observe/Review/Discovery 四场景框架，是当前阶段最值得借鉴的单点改进。
