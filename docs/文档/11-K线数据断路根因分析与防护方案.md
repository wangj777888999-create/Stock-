## 一、结论摘要

K 线数据断路**不是单一 bug，而是同一条链路上存在多个独立的断裂点**，每次迭代都可能触发其中一个。本次调查共发现 **5 类结构性脆弱点**，其中 1 类已在当前代码中形成**确认的活跃 bug**，需立即修复。

| 编号 | 类别 | 严重程度 | 当前状态 |
|------|------|---------|---------|
| F1 | `count=99999` 哨兵值与 API 上限冲突 | 🔴 严重 | ✅ 已修复（`17ff8e8`） |
| F2 | AKShare 列名硬编码，随版本漂移 | 🟠 高 | ✅ 已修复（`6f10f9b`） |
| F3 | K 线记录全 None 时后端仍返回 success:true | 🟠 高 | ✅ 已修复（`6f10f9b`） |
| F4 | 前端 `_klineChart` 未销毁就重建，ResizeObserver 累积 | 🟡 中 | ✅ 已修复（`6f10f9b`） |
| F5 | `_patch_requests` Lock 持有期间并发 K 线请求阻塞 | 🟡 中 | ✅ 已修复（`6f10f9b`） |

```

              ▼
            stock_service.py: get_kline()
              ├─ cache.get(cache_key)                  ← SQLite 缓存命中则直接返回
```

## 三、逐项根因分析

### F1 ✅ count=99999 哨兵值与 API 上限冲突（已修复 `17ff8e8`）

#### 触发时机
代码质量修复提交 `8c83157` 为 `count` 参数加了上限 `le=1000`，目的是防止 OOM 请求。

#### 证据

**前端**（`templates/index.html:3355`）：
```javascript
const count = isIntraday ? 480 : 99999;   // 日/周/月线一律发 99999
```

**路由**（`routers/stock.py:42`，8c83157 修改后）：
```python
count: int = Query(default=120, ge=1, le=1000)   // 99999 > 1000，FastAPI 返回 422
```

**实测响应**：
```json
HTTP 422
{"detail":[{"type":"less_than_equal","loc":["query","count"],
  "msg":"Input should be less than or equal to 1000","input":"99999"}]}
```

**前端处理**：
```javascript
if (!json.success || !json.data?.length) {
    container.innerHTML = '<div class="empty-state">暂无K线数据</div>';
    return;
}
```

422 响应没有 `success` 字段 → `!json.success` 为 `true` → 显示"暂无K线数据"。

#### 根本原因

`count=99999` 是一个**隐式哨兵值**，被用于表达"返回全量数据"的语义，分散在三处：

| 位置 | 代码 | 含义 |
|------|------|------|
| 前端 | `count = 99999` | "我要全量" |
| 后端路由 | `le=1000` | "最多允许1000" |
| 后端服务 | `if count < 99999:` | "99999=全量，否则切片" |

三处使用同一个魔法数字表达不同含义，任何一处变动都会断路，且**没有编译期或运行期检查**。

#### 修复方案

用显式参数代替哨兵值：

```python
# routers/stock.py
@router.get("/api/stock/kline/{symbol}")
async def api_stock_kline(
    symbol: str,
    period: str = "day",
    count: int = Query(default=120, ge=1, le=1000),
    all_history: bool = False,        # ← 新增：显式"全量"开关
    indicators: str = "",
):
    effective_count = 99999 if all_history else count
    return await stock_service.get_kline(symbol, period, effective_count, indicators=indicators)
```

```javascript
// 前端 index.html：用 all_history=true 代替 count=99999
const url = isIntraday
    ? `/api/stock/kline/${_stockCode}?period=${period}&count=480`
    : `/api/stock/kline/${_stockCode}?period=${period}&all_history=true`
      + (showIndicators ? '&indicators=rsi,macd' : '');
```

**期货路由**（`routers/market.py:56`）同样用 `count=99999`，需同步处理：
```python
# 当前
async def api_futures_kline(symbol: str, count: int = 120):

# 修复：加 all_history 参数
async def api_futures_kline(symbol: str, count: int = Query(default=120, ge=1, le=1000), all_history: bool = False):
```

### F2 🟠 AKShare 列名硬编码，随版本漂移断路

#### 触发时机
AKShare 升级（月均 2-3 次发版）偶尔会调整列名。

#### 证据

`stock_service.py` 中直接使用字符串访问 AKShare 返回的 DataFrame 列：

```python
# A股/港股 K线（stock_service.py:490-519）
"date": str(row["日期"])[:10],           # ← 直接下标，KeyError 即崩
"open": _clean(row.get("开盘")),          # ← .get() 返回 None 但不报错
"close": _clean(row.get("收盘")),
"high": _clean(row.get("最高")),
"low": _clean(row.get("最低")),
"volume": _clean(row.get("成交量")),
```

- `row["日期"]`：列名变化 → `KeyError` → 被外层 `except Exception` 捕获 → 返回 `{"success": False, "error": "获取K线失败: ..."}`
- `row.get("开盘")`：列名变化 → 返回 `None` → `_clean(None)` 返回 `None` → 前端拿到全 `None` 的 OHLC 数据 → LightweightCharts 渲染空白

#### 修复方案

加防御性列名映射，在读取 DataFrame 时验证并记录实际列名：

```python
# stock_service.py：A股/港股 K线读取段
KLINE_COL_ALIASES = {
    "date":   ["日期", "date", "Date", "交易日"],
    "open":   ["开盘", "open", "Open", "开盘价"],
    "close":  ["收盘", "close", "Close", "收盘价"],
    "high":   ["最高", "high", "High", "最高价"],
    "low":    ["最低", "low", "Low", "最低价"],
    "volume": ["成交量", "volume", "Volume", "成交量(手)"],
}

def _resolve_col(df_columns, aliases):
    """从候选列名列表中找到 DataFrame 实际存在的列名，找不到返回 None。"""
    for name in aliases:
        if name in df_columns:
            return name
    return None

def _extract_kline_row(row, col_map):
    """用已解析的列映射从行中提取 OHLCV，任一关键列缺失则返回 None。"""
    date_val = row.get(col_map["date"])
    if date_val is None:
        return None
    return {
        "date": str(date_val)[:10],
        "open": _clean(row.get(col_map["open"])),
        "close": _clean(row.get(col_map["close"])),
        "high": _clean(row.get(col_map["high"])),
        "low": _clean(row.get(col_map["low"])),
        "volume": _clean(row.get(col_map["volume"])),
    }
```

在解析前记录一次实际列名，方便排查：

```python
if df is not None and not df.empty:
    logger.debug(f"AKShare columns ({symbol}): {list(df.columns)}")
    col_map = {field: _resolve_col(df.columns, aliases)
               for field, aliases in KLINE_COL_ALIASES.items()}
    missing = [f for f, c in col_map.items() if c is None]
    if missing:
        logger.error(f"K线列名映射失败 ({symbol}): 缺少 {missing}，实际列={list(df.columns)}")
        return {"success": False, "error": f"数据源列名变更，缺少: {missing}"}
```

### F3 🟠 后端返回全 None 的 records 仍视为成功

#### 触发时机
F2 的 `.get()` 路径（静默返回 None）触发时。

#### 证据

```python
# stock_service.py:562-566
resp = {"success": True, "data": records}
cache.set(cache_key, resp, TTL_DAILY)   # 将全 None 的数据写入缓存！
return resp
```

全 None 数据被缓存 300 秒。即使 AKShare 恢复，缓存内的坏数据仍会持续返回 5 分钟。

#### 修复方案

写缓存前验证数据有效性：

```python
# stock_service.py：组装 records 后、cache.set 前
valid_records = [r for r in records
                 if r.get("close") is not None and r.get("open") is not None]
if not valid_records:
    logger.error(f"K线记录全部无效 ({symbol})，不写缓存，返回错误")
    return {"success": False, "error": "K线数据解析失败，所有OHLC字段为空"}

resp = {"success": True, "data": valid_records}
cache.set(cache_key, resp, TTL_DAILY)
return resp
```

### F4 🟡 前端 `_klineChart` 未销毁就重建，ResizeObserver 泄漏

#### 触发时机
用户切换周期（日→周→月→日…），每次 `loadKline()` 在旧图实例上直接创建新图。

#### 证据

```javascript
// templates/index.html:3361（当前）
_klineChart = LightweightCharts.createChart(container, {...});
// 没有 _klineChart.remove() 调用！旧实例仍在内存中

// 3410：每次都创建新的 ResizeObserver，从不解绑
new ResizeObserver(() => {
    if (_klineChart) _klineChart.applyOptions({...});
}).observe(container);
```

多次切换后，`container` 元素上叠加了多个 ResizeObserver，每次窗口大小变化都触发所有旧实例的回调。旧回调中 `_klineChart` 已指向最新实例，但旧 ResizeObserver 实例永远不会被 GC，内存持续增长，最终可能导致渲染卡顿。

#### 修复方案

在 `loadKline` 入口处先销毁旧图实例，并保存 ResizeObserver 引用：

```javascript
let _klineResizeObserver = null;

function _disposeMainChart() {
    if (_klineResizeObserver) {
        _klineResizeObserver.disconnect();
        _klineResizeObserver = null;
    }
    if (_klineChart) {
        try { _klineChart.remove(); } catch(e) {}
        _klineChart = null;
        _klineSeries = null;
        _klineVolSeries = null;
    }
}

async function loadKline(period) {
    if (!_stockCode) return;
    _disposeMainChart();        // ← 先销毁旧图
    _disposeKlineSubCharts();
    // ...其余不变...
    _klineResizeObserver = new ResizeObserver(() => {
        if (_klineChart) _klineChart.applyOptions({ width: container.clientWidth });
    });
    _klineResizeObserver.observe(container);
}
```

### F5 🟡 `_patch_requests` threading.Lock 阻塞并发 K 线请求

#### 触发时机
`8c83157` 质量修复后，`patch_requests()` 加了 `threading.Lock`。用户快速切换周期时，多个 `asyncio.to_thread` 调用会竞争同一个锁。

#### 证据

```python
# http_client.py（8c83157 后）
_patch_lock = threading.Lock()

def patch_requests(func, **kwargs):
    with _patch_lock:              # 持锁期间其他线程阻塞
        orig_get = _requests.get
        orig_post = _requests.post
        _requests.get = session.get
        _requests.post = session.post
        try:
            return func(**kwargs)  # AKShare 网络请求，可能 5-10s
        finally:
            _requests.get = orig_get
            _requests.post = orig_post
```

AKShare 请求耗时 5-10 秒时，其他所有 K 线请求在线程池中排队等待，前端无响应。

#### 修复方向

Lock 的存在是正确的（防止并发覆盖），但 AKShare 请求本身应该超时可控：

```python
# stock_service.py：加 asyncio 超时包装
try:
    df = await asyncio.wait_for(
        asyncio.to_thread(_patch_requests, ak.stock_zh_a_hist, **kwargs),
        timeout=15.0  # 超时15秒，不让 Lock 被长期占用
    )
except asyncio.TimeoutError:
    logger.warning(f"AKShare 请求超时 ({symbol})")
    return {"success": False, "error": "行情数据源请求超时，请稍后重试"}
```

## 四、迭代安全检查清单

每次提交涉及以下文件时，必须过一遍此清单：

### 修改 `routers/stock.py` 时

- [ ] `count` 参数的上限（`le=N`）是否与前端发送的 `count` 最大值兼容？
- [ ] 新增路由参数后，是否检查前端所有 `fetch('/api/stock/kline/...')` 的 URL 构造？
- [ ] `period` 参数是否有白名单校验？（目前没有，传 `period=foo` 会走 `period_map.get("foo", "daily")` 降级）

### 修改 `stock_service.py` 时

- [ ] 新增 AKShare API 调用后，是否记录了列名访问方式（中文 vs 英文）？
- [ ] DataFrame 行访问是否用 `row["列名"]`（KeyError 风险）还是 `row.get("列名")`（静默 None 风险）？
- [ ] 组装 records 后是否验证 OHLC 字段不全为 None？

### 修改 `templates/index.html` 时

- [ ] `loadKline` 中是否调用了 `_disposeMainChart()`？
- [ ] 新增 ResizeObserver 是否保存引用并在下次调用时 `disconnect()`？
- [ ] `fetch` URL 中的 query 参数是否与后端路由参数名一致？

### 升级 AKShare（`pip install -U akshare`）时

- [ ] 运行 `python -c "import akshare as ak; df = ak.stock_zh_a_hist(...); print(list(df.columns))"` 确认列名未变
- [ ] 运行 `python -c "import akshare as ak; df = ak.stock_hk_hist(...); print(list(df.columns))"` 同上
- [ ] 检查 AKShare changelog 是否有 breaking change 标注
