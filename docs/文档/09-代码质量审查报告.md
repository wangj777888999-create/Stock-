## 二、逐文件详细发现

### 2.1 `http_client.py`（44 行）

**🔴 严重 — patch_requests() 并发竞态**

`patch_requests()` 通过临时覆盖 `requests.get / requests.post` 全局函数来绕过代理：

```python
orig_get = _requests.get
_requests.get = session.get   # ← 另一协程此刻读到的是 session.get
try:
    return func(**kwargs)
finally:
    _requests.get = orig_get  # ← 恢复的是错误的 orig_get
```

若两个协程同时调用此函数，`orig_get` 会互相覆盖，导致后续所有 `requests.get` 调用都使用了错误的会话或丢失代理绑定。

**修复方向**：改用 `requests.Session` 传参而不是全局覆盖，或加 `asyncio.Lock`。

### 2.2 `routers/blogger.py`（当前约 700 行，含 ae52a30 新增）

#### 🔴 严重 — WebSocket 接收 URL 未验证（第 483 行）

```python
urls = data.get("urls", [])
```

URLs 直接传入 Playwright 爬取，Playwright 会遵从协议：

- `file:///C:/Windows/System32/...` — 读取本地任意文件
- `http://192.168.x.x/admin` — 内网穿透
- `javascript:alert(document.cookie)` — JS 执行

> ⚠️ 行号已随 `ae52a30` 从 469 更新为 483，问题本身未修复。

**修复方向**：白名单校验协议（只允许 `https://mp.weixin.qq.com/...`）。

#### 🔴 严重 — Cookie 明文写入 JSON（第 396–420 行）

```python
cookies = await context.cookies()
cookie_str = "; ".join(
    f"{c['name']}={c['value']}"
    for c in cookies
    if "qq.com" in c.get("domain", "")
)
# ...
CONFIG_FILE.write_text(json.dumps(save_data, ...), encoding="utf-8")
```

整个 `mp.weixin.qq.com` 域下的 Cookie Jar 以明文存储到 `user_config.json`，文件无访问权限限制。获得文件读权限即可冒充已登录用户。

**修复方向**：仅提取必要的 `mp_cookie`/`mp_token` 字段；或者不落盘，只保存在内存（`state.py`）并标注"重启后需重新扫码"。

#### 🟠 高 — asyncio.gather 所有任务失败无感知

```python
results = await asyncio.gather(*tasks, return_exceptions=True)
```

`return_exceptions=True` 时，异常作为普通返回值，后续只检查 `isinstance(r, dict)`，但未处理 `isinstance(r, Exception)` 的分支。所有任务抛异常时，前端只会收到空结果，无错误提示。

#### 🟠 高 — Semaphore 在每次 WebSocket 调用中新建

每次连接建立都创建新的 `asyncio.Semaphore`，连接中途断开时旧 Semaphore 无法被 GC（有 Task 持有引用），积累内存泄漏。

### 2.4 `analyzer.py`（300 行）

#### 🟠 高 — asyncio.gather 崩一停全（第 111 行）

```python
results = await asyncio.gather(*tasks, return_exceptions=False)
```

`return_exceptions=False`（默认值）表示任一 persona 分析失败就取消其余所有任务。若 AI API 偶发超时，整篇文章的分析结果全部丢失。

**修复方向**：改为 `return_exceptions=True`，对每个结果单独检查是否为异常。

#### 🟠 高 — AI 响应直接下标访问（第 186 行）

```python
data["choices"][0]["message"]["content"]
```

若 API 返回 `choices=[]`（配额耗尽、内容过滤等），直接 `IndexError`，未被内层 try/except 覆盖。

#### 🟡 中 — 神函数 analyze()（第 34–79 行）

单函数 46 行，混合了：文本拼装、prompt 构造、HTTP 调用、降级逻辑、响应解析。职责边界不清晰。

### 2.5 `database.py`（222 行）

#### 🟠 高 — 迁移静默吞错（第 209–211 行）

```python
except sqlite3.OperationalError:
    pass
```

"列已存在" 和 "磁盘只读" 产生的错误码相同，两者都被忽略。若磁盘满导致迁移失败，应用启动时没有任何报警，数据库结构与代码不一致。

**修复方向**：检查错误消息是否含 `"duplicate column name"`，只忽略此类错误。

#### 🟡 中 — 无迁移版本记录

每次启动都重跑所有 `ALTER TABLE`，依赖 OperationalError 做幂等。正确做法是维护一张 `schema_version` 表，记录已应用的迁移序号。

#### 🟡 中 — WAL 文件无限增长风险

WAL 模式未设置 `journal_size_limit` 或手动触发 `PRAGMA wal_checkpoint(TRUNCATE)`，长时间运行后 `-wal` 文件可能达到数百 MB。

### 2.7 `scraper.py`（235 行）

#### 🟠 高 — cleanup() 存在但未接入 app.py 的 shutdown 钩子 ⟳ 已局部改善

> **ae52a30 变动**：`scraper.py` 新增了 `cleanup()` 方法，代码如下：
>
> ```python
> async def cleanup(self):
>     if self.context:
>         await self.context.close()
>     if self.browser:
>         await self.browser.close()
>     if self.playwright:
>         await self.playwright.stop()
> ```
>
> ✅ `cleanup()` 逻辑本身正确。
> ❌ 但 `app.py._shutdown()` 目前**未调用** `scraper.cleanup()`：
>
> ```python
> @app.on_event("shutdown")
> async def _shutdown():
>     try:
>         stop_scheduler()
>     except Exception:
>         pass
>     close_db()   # ← 缺少 await scraper.cleanup()
> ```
>
> 应用正常退出时 Playwright 进程仍会残留。

**修复方向**：在 `_shutdown()` 中补加：

```python
try:
    await scraper.cleanup()
except Exception:
    pass
```

#### 🟠 高 — 多层 `except: pass` 吞掉 Playwright 错误（第 101、109、111 行）

```python
except Exception:
    pass
```

`wait_for_selector` 超时 vs 页面崩溃 vs 选择器不存在 — 三种错误都被静默丢弃，调试时无法定位失败原因。

#### 🟡 中 — 内联 JS 大段字符串（第 117–166 行）

50 行 JavaScript 以多行字符串形式内嵌 Python，既无语法高亮，也无法单独测试。应提取到 `scraper_extract.js` 文件，运行时加载。

## 三、整体架构级发现

### 3.1 缺少请求验证层

全部 15 个路由文件中，仅 `verify.py` 使用了 Pydantic 模型做入参验证（且模型字段无格式约束）。其余路由直接接收原始 Query/Path 参数，未做：

- 长度上限（防止大请求体）
- 字符集验证（防止特殊字符注入）
- 范围验证（防止 OOM 型参数）

**建议**：所有写接口改为 Pydantic 模型 + `Field(..., max_length=..., ge=..., le=...)`。

### 3.2 神类集中度过高

| 文件 | 行数 | 主要职责数 |
|------|------|-----------|
| `blogger.py` | 724 | 7 |
| `stock_service.py` | 909 | 8+ |
| `routers/blogger.py` | ~700 | 5 |
| `templates/index.html` | 4594+ | 9 页面 |

单文件过长导致：修改影响面不可预测、难以单元测试、AI 辅助开发时上下文超限。

### 3.3 错误处理不一致

全库共有 **23 处** `except Exception: pass` 或 `except Exception as e: logger.warning(...)` 后不重新抛出。错误被吞掉后，上层调用者拿到的是 `None` 或空列表，无法区分"无数据"和"出错"两种状态。

### 3.4 测试覆盖率为零

项目目前无任何测试文件（unit / integration / e2e）。鉴于项目定位为"个人投资自用工具 + 求职作品集"，短期可接受，但若涉及实盘交易逻辑，至少应为评分算法和 CSV 导入逻辑编写基本 happy path 测试。
