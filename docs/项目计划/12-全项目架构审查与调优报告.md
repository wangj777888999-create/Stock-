## 二、整体评价

**这是一个相当成熟、迭代活跃的个人级项目，工程质量在"个人自用本地工具"这个定位下明显高于平均水平**：

✅ **亮点**
- `DataRouter`（services/data_router.py）是一处教科书级的"最小内核"抽象：契约注册 + EWMA 延迟排序 + 并发 race + stale 兜底 + 健康面板，业务层一行接入。注释克制、职责单一。
- 两层缓存（L1 进程内 dict + L2 SQLite 持久化）设计清晰，TTL 分级合理。
- 多市场覆盖广（A/港/美/韩/日/期货/基金/加密），且每类都有降级源。
- 配置安全策略明确：敏感字段（密码/Cookie/Token/api_key）**只从环境变量读取，永不落 JSON 文件**——这一点贯彻得很好。
- 数据库迁移用 `CREATE TABLE IF NOT EXISTS` + 忽略 `duplicate column` 的幂等 ALTER，存量升级安全。

⚠️ **核心债务**：快速迭代留下的"新抽象已建、旧实现未清"的中间态，以及文档/CLAUDE.md 与代码脱节。详见下。

## 三、问题清单（按严重度分级）

### 🔴 P0 — 应尽快处理（低风险、明确收益）

**3.1 三套数据降级机制并存，迁移未完成**
- `DataRouter.fetch`（新，正确方向）— 已用于 stock_service / signal_service / cockpit_service / routers/health
- `services/source_racer.race_sources`（旧）— stock_service.py 仍有 **4 处**调用（公司信息、财务、L660/758/908/1040）
- `stock_utils.call_with_fallback`（更旧）— **全仓 0 调用点，纯死代码**

> 风险：三套语义相近的并发降级逻辑，新人/未来的你需要分别理解；死代码误导阅读。
> 建议：① 立即删除 `call_with_fallback`（死代码）；② 把 stock_service 里 4 处 `race_sources` 逐步迁到 DataRouter 契约（公司信息/财务可新增 `company_profile`、`financial` 契约），迁完后删除 `source_racer`。

**3.2 仓库里提交了 436KB 的源码压缩包**
- `wexin-read-mcp-main.zip`（436KB）已被 `git ls-files` 跟踪——这是项目最初的归档包，不应进版本库。
- 建议：`git rm --cached wexin-read-mcp-main.zip` 并加入 `.gitignore`。

**3.3 根目录杂物污染工作区**
- `data.db`（根目录，0 字节残留；真实库在 `wexin-read-mcp-main/data.db` 5MB，已正确 gitignore）
- `未命名.md`（空）、`未命名.base` / `未命名 1.base` / `未命名 2.base`（各 40 字节，疑似 Obsidian 误建）
- `行业调研对比说明.html`（13KB）、`行研skill.docx`（236KB）、`StockPulse-AI构建成果.md`
- 建议：删除空文件与 `.base`；把散落的 html/docx 归入 `docs/` 或删除；`.gitignore` 增补 `*.base`、`未命名*`。

### 🟠 P1 — 架构调优（中等收益，需评估）

**3.4 循环依赖导致大量函数内局部 import**
- 实测函数内 `import` 数量：stock_service 20、sector_service 10、providers 8、app 8、blogger 7。
- 这是典型"模块互相引用 → 只能在函数体内延迟导入"的规避手法。能跑，但说明模块边界不清（尤其 stock_service ↔ services/providers ↔ signal_service 三角）。
- 建议：不必激进重构。中期可把被反复延迟导入的纯函数（如 `_parse_tencent_quote`、`get_exchange`）下沉到无依赖的工具模块，斩断环。

**3.5 `stock_service.py` 1151 行，职责过载**
- 单文件混合：A/港/美报价、K 线、公司信息、财务、资金流、批量行情、技术指标拼装……
- 建议：按"报价 / K线 / 基本面（公司+财务） / 资金流"拆为 4 个子模块，或下沉到 `services/` 契约。配合 3.1 的迁移一并做。

**3.6 异常吞噬，可观测性弱**
- 裸 `except Exception` 计数：cockpit_service 20、blogger 14、providers 12、sector_service 11、stock_service 10。
- providers 里多数 `except Exception: return None` 是**有意的**（降级源失败即放弃），但**不记日志**——线上某个源默默全挂时无从察觉。DataRouter 的 `_record_failure` 已记 `last_error`，但 provider 内部 try 把异常吞了，外层拿不到真实错误。
- 建议：provider 内 `except` 至少 `logger.debug(...)` 记一行；或干脆**不在 provider 内 try**，让异常抛给 DataRouter 统一记录（DataRouter 本就会捕获并计入统计）。

**3.7 FastAPI `@app.on_event` 已弃用**
- app.py L125/L192 用 `on_event("startup"/"shutdown")`，新版 FastAPI 推荐 `lifespan` 上下文管理器。
- 建议：迁移到 `lifespan`，顺手把 7 个 `asyncio.create_task` 预热任务集中管理（便于关闭时取消）。

### 🟡 P2 — 可选优化（低优先）

**3.8 SQLite 单连接 + 全局写锁**
- `database.py` 单 `_db` 连接 + `write_lock` 串行化写。调度线程与请求线程共享一条连接。
- 对个人本地单用户场景**完全够用**，不必改。仅记录为已知瓶颈：若未来并发上来，考虑连接池或每线程连接。

**3.9 前端单文件 8923 行 / 437KB**
- 已有 `docs/项目计划/09-前端重构改进方案.md` 跟踪（Phase 1-2 已完成）。继续推进既有方案即可，本报告不重复。

**3.10 缺测试与 CI**
- 仅有 3 个 benchmark 脚本，无单测、无 CI。个人项目可接受，但 DataRouter / stock_utils 缓存 / 代码解析（stock_utils.detect_market 等纯函数）是高价值、易测的单测切入点。
